## Task 4 — Code-level security review

**Done.** Verified the following facts via grep and file reads:

1. **Frontend never touches Supabase directly.**
   - `grep -r "@supabase/supabase-js\|supabase.from\|createClient" src/src/` → zero matches
   - `src/src/utils/api.ts` is a plain `fetch` wrapper against `/api/*`
   - `src/src/stores/authStore.ts` is a Zustand store holding `{token, user}`; no Supabase client
   - All frontend data access goes through the Express backend

2. **Backend uses `service_role` exclusively.**
   - All 19 `.from(...)` calls across `api/src/routes/` use `supabaseAdmin` (service_role)
   - The anon `supabase` client exported from `api/src/utils/supabase.ts` is **never imported anywhere** (verified by grep)
   - Distribution: `leagues.ts` (3), `notifications.ts` (2), `commissioner.ts` (3), `admin.ts` (5), `debug.ts` (6), plus many more queries inside `cards.ts` via variable references

3. **GRANT audit.** The six card-related tables (`cards`, `played_cards`, `user_cards`, `switcheroo_log`, `user_switcheroo`, `weekly_card_picks`) have grants ONLY for `service_role`. Neither `anon` nor `authenticated` can touch them via PostgREST today, with or without RLS.

4. **No race conditions.** The previous agent's "frontend race condition" theory for the intermittent 403s was wrong. The intermittency came from some queries using service_role (which worked) and others using authenticated (which failed due to missing grants). Once grants were fixed, 403s stopped entirely. Verified.

**Implication:** RLS on public tables is currently defense-in-depth only. No functional traffic depends on policy evaluation. This is what makes the "enable RLS with no policies" recommendation for the card tables safe.

## Task 5 — Duplicate policy cleanup & RLS posture

### 5a. `chat_messages` duplicates — FIXED ✓

**Problem:** Migration `002_rls_policies.sql` created `chat_select_member` and `chat_insert_member`. Migration `008_phase4_polish.sql` then added `chat_messages_select` and `chat_messages_insert` with inline subqueries doing the same thing.

I verified the two policy pairs were semantically equivalent by reading `is_league_member()`'s definition:

```sql
-- From 002_rls_policies.sql
CREATE FUNCTION is_league_member(p_league_id UUID) RETURNS BOOLEAN AS $$
BEGIN
  RETURN EXISTS (
    SELECT 1 FROM teams
    WHERE league_id = p_league_id AND user_id = auth.uid()
  );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

The inline subquery in the 008 policies (`league_id IN (SELECT t.league_id FROM teams t WHERE t.user_id = auth.uid())`) is identical. Dropping the duplicates is a no-op.

**Applied:** Migration `012_cleanup_duplicate_chat_policies.sql` drops `chat_messages_select` and `chat_messages_insert`. Verified — `chat_messages` now has exactly `chat_select_member` and `chat_insert_member`.

### 5b. Card tables RLS hardening — DRAFT READY, PENDING YOUR REVIEW ⚠️

**Problem:** Six tables have RLS disabled; three have orphaned policies (the advisor ERRORs above).

**Draft migration:** `db/migrations/013_enable_rls_on_card_tables.sql` — **not yet applied.**

Strategy:
1. Drop the four orphaned policies (`cards_all_admin`, `cards_select_authenticated`, `played_cards_select`, `user_cards_select_own`) — they reference `auth.uid()` and were written for a direct-client-access model that the app doesn't use.
2. Enable RLS on all six tables.
3. Create no new policies. `service_role` bypasses RLS, backend continues to work exactly as today, anon/authenticated are double-locked (no grants AND no policies).

Risk: zero. Verified three ways (backend uses service_role only, frontend never touches Supabase, anon/authenticated have no grants anyway).

**Action needed:** Read the migration file, decide yes/no, and either I apply it or we discard it. If you want some of these tables directly queryable by authenticated users in the future (e.g., let a user query their own `user_cards` without going through the backend), don't apply 013 as-is — we'd need explicit SELECT policies instead.

### 5c. `users_insert` RLS policy WITH CHECK (true) — DECISION NEEDED

The advisor flags `public.users.users_insert` as `rls_policy_always_true`. The policy allows any INSERT into `public.users` with no restrictions.

If this is intentional to allow the signup flow to create user rows client-side after Supabase Auth signup, it's fine — though the more conventional approach is a trigger on `auth.users` that creates the public row server-side, letting you remove the public INSERT policy entirely.

If it's not intentional, we should tighten it to `WITH CHECK (id = auth.uid())`.

**Action needed:** Confirm whether the signup flow relies on this policy. I can check the backend code and propose a fix either way if you want.
